# One-shot implementation plan: fix dry-run/live repeated markets + real continuous scanning

## Final goal

Convert the current app from this:

```txt
Dry-run/live simulation = hardcoded repeated fake markets
```

To this:

```txt
Dry-run/paper mode = real Polymarket/Kalshi data + simulated execution
Live mode = real Polymarket/Kalshi data + real execution later
Demo mode = hardcoded fake templates only for testing UI
```

The repeated markets like Bitcoin/Solana/NBA/global-temperature must stop appearing again and again unless they are real unique markets from a venue.

---

# 1. Required mode separation

Create three clear modes:

```txt
DEMO
PAPER
LIVE
```

## DEMO mode

Purpose:

```txt
Use mock MARKET_TEMPLATES only for UI/demo testing.
```

Allowed:

```txt
Hardcoded markets
Fake liquidity
Fake spread
Fake probability
Fake execution
```

Not allowed:

```txt
Showing as real dry-run/live trading
```

---

## PAPER mode

Purpose:

```txt
Use real venue market data, but simulate orders.
```

Allowed:

```txt
Real Polymarket data
Real Kalshi data
Real market snapshots
Real research
Real triage
Real risk decision
Simulated paper order
Simulated paper PnL
```

Not allowed:

```txt
Hardcoded MARKET_TEMPLATES
Real money execution
Instant fake duplicate orders
```

---

## LIVE mode

Purpose:

```txt
Use real data and real execution connectors.
```

For now:

```txt
LIVE mode should be disabled behind safety flag until real order connectors exist.
```

# 6. Market-loop algorithm

Create:

```txt
src/lib/engine/market-loop.ts
```

## Loop frequency

```txt
Full scan: every 1–5 minutes
Deep reprocessing: cooldown-based
Snapshot refresh: every scan
Research: only if candidate passes score
```

## Loop logic

```txt
1. Read trading mode from DB.
2. If mode = DEMO:
   use demo simulation only.
3. If mode = PAPER:
   use real venue scanners.
4. If mode = LIVE:
   use real venue scanners + live execution only if safety enabled.
5. Run scan for every enabled venue.
6. Upsert markets.
7. Create snapshots.
8. Score markets.
9. Deduplicate.
10. Apply cooldown.
11. Queue eligible candidates.
12. Log scan result.
```

## Thresholds

```txt
score < 50:
  skip

70–84:

85+:
  triage + research
```

# 9. Cooldown rules

Add these rules:

```txt
Same market triage cooldown: 1–3 hours
Same market research cooldown: 6–24 hours
Same market execution cooldown: until major price move or manual reset
Same market WATCH cooldown: recheck every 1–6 hours
Failed research retry: max 3 retries with backoff
```

Reprocess only if:

```txt
implied probability moved by 3%–5%
liquidity increased significantly
spread improved significantly
new venue snapshot shows meaningful change
manual force-research clicked
cooldown expired
```

# 13. Pipeline update plan

## Current issue

```txt
Research failure can stop pipeline too early.
WATCH creates fake orders.
Risk exposure uses wrong fields.
Orders are instantly filled.
```

## Required pipeline behavior

```txt
1. Load market and latest snapshot.
2. Run triage.
3. If triage = irrelevant, mark skipped.
4. If relevant, run adaptive research.
5. If research provider fails, fallback.
6. Continue to judge if minimum research exists.
7. Run risk.
8. If risk = SKIP, save decision only.
9. If risk = WATCH, create Watchlist record only.
10. If risk = BID, create paper/live execution job.
```

# 14. Adaptive research plan

## Research levels

```txt
NONE:
  skip

QUICK:
  basic venue data + one search provider

STANDARD:
  search + source summary + LLM judge

DEEP:
  search + Firecrawl + DeerFlow/TradingAgents + debate + judge
```

## Selection rule

```txt
candidateScore 70–84:
  QUICK

candidateScore 85–89:
  STANDARD

candidateScore 90+:
  DEEP
```

## Fallback rule

If DeerFlow fails:

```txt
fallback to SearXNG + Firecrawl/basic search
continue pipeline
mark provider failure in logs
do not leave market stuck forever
```
